## Anti-Patterns Verdict

**Does this look AI-generated? No.** Both the design review and the deterministic scan agree this is disciplined, intentional work.

**LLM assessment:** Zero decorative gradients, no `background-clip:text`, no glassmorphism, no hero-metric template, no identical card grids, no uppercase eyebrows. The Silent-Until-State, One-Accent, Machine-Truth, and Flat-By-Default rules from DESIGN.md are enforced in the code. Specialized visualizations (24-bar reachability quorum, status-badge pills) show genuine domain expertise rather than template reflex.

**Deterministic scan (`detect.mjs`, 51 findings):**
- `design-system-font-size` × 26 (advisory) — literal font-sizes off the DESIGN.md type ramp, almost all in `styles.css`, 2 in `App.css`.
- `design-system-radius` × 21 (advisory) — radii (4px, 5px, 6px, 11px…) outside the documented `sm:10 / md:16 / pill:999` scale.
- `design-system-color` × 3 (advisory) — `rgba(5,8,16,.5/.6)` scrim/backdrop overlays not in the token list. **Likely false positives**: these are modal/rail scrim washes, a legitimate role the token set simply doesn't name.
- `layout-transition` × 1 (warning) — `transition: width` on the rail resizer (styles.css:287). Real, but the rail width is drag-driven and deliberate; low impact.

The scan found no true slop — only design-system drift: the code uses more type sizes and radii than DESIGN.md documents. Either snap to tokens or expand the documented scale.

## Overall Impression

This is an operator's instrument panel that already respects its user. It reads state plainly, reserves color and motion for signal, and keeps a consistent vocabulary screen to screen. The single biggest opportunity is the failure path: when DNS *doesn't* update, the UI currently says "failed" and then forgets — exactly the moment this audience most needs the truth.

## What's Working

1. **The status-badge system.** A 7px dot + label pill carries complete per-domain health in minimal space; the pulsing "updating" dot and blue card border are a clean two-layer live signal.
2. **The 24-bar reachability quorum chart.** Dense, legible, semantic-colored, with a glowing live bar — specialized operator visualization, not marketing filler.
3. **True light/dark parity.** Every token has a mirror; the backdrop-blur topbar uses `color-mix` so the theme reads authentically in both modes. Not a bolted-on dark mode.

## Priority Issues

- **[P1] Error recovery is a dead end.** A failed sync/hook shows a 3.2s toast ("Sync failed") with no reason, no retry, no link. This audience reads logs — give them the thread. *Fix:* persistent error toast with a "View in Logs" link (auto-filtered to ERROR) and a "Retry", plus a one-line last-error under the domain card's status badge. *Command:* /impeccable harden
- **[P1] No confirmation on destructive delete.** One click removes a domain; red hover ≠ confirmation. Accidental deletes are costly in self-hosted infra. *Fix:* confirm dialog naming the host, or an "Undo" toast (5s) on delete/disable. *Command:* /impeccable harden
- **[P2] First-run onboarding & help absent.** No glossary for provider / IP source / hooks, no example hostname placeholder, empty states don't teach. Operators cope; new self-hosters stall. *Fix:* teaching empty states, inline field help in the Add-Domain modal, term tooltips. *Command:* /impeccable onboard
- **[P2] No power-user efficiency paths.** No command palette / keyboard nav, no bulk domain operations, log search is text-only (no filter by domain/provider). *Fix:* Cmd/Ctrl+K palette, keyboard nav in the domain grid, structured log filters. *Command:* /impeccable shape
- **[P3] Interval selector = 5 chips, no default guidance.** Exceeds the ≤4 working-memory guideline and forces an uncontextualized choice. *Fix:* trim to 3-4 common values + Custom, sensible default, one-line trade-off hint. *Command:* /impeccable clarify

## Persona Red Flags

**Power User (homelab operator, reads logs, wants density + keyboard):**
- No keyboard shortcuts / command palette.
- No bulk operations (multi-select toggle / delete / re-sync).
- Log search is text-only — can't filter by domain or provider.
- No saved views/filters ("all errored domains", "last 6h").
- Respected: resizable/collapsible rail; monospace IPs & logs.

**First-Timer (small-org sysadmin new to self-hosted DNS):**
- No glossary — "provider", "IP source", "hooks" are unexplained.
- Add-Domain modal has blank fields, no example hostname.
- Cryptic failures — "Sync failed" with no why.
- Settings assume knowledge (startup update, interval, notifications).
- Helped: intuitive status colors; empty states exist (if thin).

## Minor Observations

- Design-system drift: `styles.css`/`App.css` carry font sizes and radii off the DESIGN.md ramp/scale. Snap to tokens or expand the documented scale so the system stays the source of truth.
- Provider indigo badge is dim in dark mode — check hover contrast.
- Settings groups (Scheduling / Behavior / IP Source) lack dividers on the 2-col grid; grouping is ambiguous.
- "Next check" countdown shows "—" with no loading state when the backend is slow.
- Hook event tags are correctly monospace but hard to scan for newcomers — tooltip them.

## Questions to Consider

1. Is the dense Overview the right landing page for a first-timer, or should first-run be a task picker?
2. Should "Provider" be progressively disclosed rather than a first-class step in the Add-Domain flow?
3. Is "Synced" the right word, or does it collide with DNS-replication meaning? ("Up to date"/"Current"?)
4. Should the topbar IP pills be persistent, or only surface on change?
5. Any need for log retention/export for the small-org compliance case?
